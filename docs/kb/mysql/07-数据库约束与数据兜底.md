# 07-数据库约束与数据兜底

数据库约束用于保证数据入库前的最后底线。

在真实项目里，数据校验通常不是只靠一层完成，而是分三层：

```text
DTO / Validation：管格式和基础必填
Service：管业务规则
数据库约束：管最终兜底
```

一句话：

```text
Validation 管格式，Service 管业务，数据库管底线。
```

---

## 1. 为什么需要数据库约束

例如部门名称不能重复。

Service 层可以提前判断：

```java
if (mapper.existsByName(request.getName())) {
  throw new IllegalArgumentException("department name already exists");
}
```

这能给前端更友好的错误。

但它不是最终保证。

极端情况下，两个请求同时进来：

```text
请求 A 检查 name 不存在
请求 B 检查 name 不存在
请求 A 插入成功
请求 B 也插入
```

如果数据库没有唯一约束，就可能出现重复数据。

所以真实项目通常会做两层保护：

```text
Service 提前判断：用户体验更好
数据库唯一约束：最终保证数据不脏
```

---

## 2. 主键约束 primary key

主键用于唯一标识一条记录。

```sql
id int primary key auto_increment
```

作用：

```text
id 不能为空
id 不能重复
可以通过 id 精确定位一条数据
```

常见理解：

```text
id 是技术唯一标识
name / phone / email 是业务唯一字段
```

---

## 3. 非空约束 not null

非空约束表示字段不能是 `null`。

```sql
name varchar(50) not null
```

作用：

```text
插入数据时，name 不能为 null
```

注意：

```text
not null 不能阻止空字符串 ''
```

所以字符串必填通常还要在 DTO 上写：

```java
@NotBlank(message = "name cannot be blank")
private String name;
```

---

## 4. 唯一约束 unique

唯一约束表示某个字段不能重复。

```sql
alter table department add unique index uk_department_name (name);
```

含义：

```text
department 表里的 name 不能重复
索引名字叫 uk_department_name
```

常见使用场景：

```text
用户名不能重复
手机号不能重复
邮箱不能重复
部门名称不能重复
订单号不能重复
身份证号不能重复
```

示例：

```sql
alter table user add unique index uk_user_phone (phone);
alter table user add unique index uk_user_email (email);
```

---

## 5. 默认值 default

默认值表示新增数据时，如果没有传该字段，数据库自动给默认值。

```sql
status int default 1
```

含义：

```text
新增时不传 status，数据库默认写入 1
```

常见场景：

```text
status 默认启用
deleted 默认未删除
sort 默认 0
create_time 默认当前时间
```

---

## 6. 字段长度限制

字段类型本身也会限制数据。

```sql
name varchar(50)
phone varchar(20)
email varchar(100)
```

含义：

```text
name 最多 50 个字符
phone 最多 20 个字符
email 最多 100 个字符
```

但真实项目里一般不只靠数据库长度限制，而是在 DTO 提前校验：

```java
@Size(max = 50, message = "name length must <= 50")
private String name;
```

这样错误更友好。

---

## 7. check 约束

`check` 用于限制字段取值范围。

```sql
status int check (status in (0, 1))
```

含义：

```text
status 只能是 0 或 1
```

注意：

```text
MySQL 8 开始才真正支持比较完整的 check 约束
```

真实项目里，状态值通常也会在 Service 或枚举中校验：

```java
if (status != 0 && status != 1) {
  throw new IllegalArgumentException("status must be 0 or 1");
}
```

---

## 8. 外键约束 foreign key

外键用于保证关联数据存在。

例如员工属于某个部门：

```sql
department_id int
```

可以加：

```sql
foreign key (department_id) references department(id)
```

作用：

```text
employee.department_id 必须在 department.id 中存在
```

但很多业务系统不强依赖数据库外键，而是在 Service 层处理关联校验。

常见原因：

```text
部署和迁移更灵活
跨库分表时外键不好用
复杂业务删除规则不完全等于数据库外键规则
降低数据库强约束带来的维护成本
```

所以要理解：

```text
外键是数据库能力，但真实项目不一定都用数据库外键。
```

## 9. 邮箱、手机号、字符格式放哪里校验

邮箱格式更适合 DTO 校验：

```java
@NotBlank(message = "email cannot be blank")
@Email(message = "email format error")
private String email;
```

手机号格式也更适合 DTO 校验：

```java
@NotBlank(message = "phone cannot be blank")
@Pattern(regexp = "^1[3-9]\\d{9}$", message = "phone format error")
private String phone;
```

手机号和邮箱不能重复，则适合：

```text
Service 提前查重
数据库 unique 最终兜底
```

Service：

```java
if (mapper.existsByEmail(email)) {
  throw new IllegalArgumentException("email already exists");
}
```

数据库：

```sql
alter table user add unique index uk_user_email (email);
```

---

## 10. 三层校验分工

| 规则 | DTO / Validation | Service | 数据库 |
| --- | --- | --- | --- |
| name 必填 | `@NotBlank` | 可补充判断 | `not null` |
| name 不重复 | 不适合 | `existsByName` | `unique index` |
| email 格式 | `@Email` | 不适合 | 不适合 |
| phone 格式 | `@Pattern` | 不适合 | 不适合 |
| status 只能 0/1 | 可自定义校验 | 常用 | 可用 `check` |
| id 唯一 | 不适合 | 一般不需要 | `primary key` |
| 外键是否存在 | 不适合 | 常用 | 可用 `foreign key` |

## 11. 数据库异常兜底

当数据库唯一约束拦截重复数据时，Spring 可能抛：

```java
DuplicateKeyException
```

可以在全局异常处理里兜底：

```java
import org.springframework.dao.DuplicateKeyException;

@ExceptionHandler(DuplicateKeyException.class)
public ApiResponse<Void> handleDuplicateKey(DuplicateKeyException e) {
  return ApiResponse.fail(400, "data already exists");
}
```

作用：

```text
即使 Service 校验漏掉，数据库报错也不会直接返回 500。
```

## 12. Department 示例

表结构：

```sql
create table department (
  id int primary key auto_increment,
  name varchar(50) not null,
  description varchar(255),
  status int default 1
);
```

给 `name` 加唯一约束：

```sql
alter table department add unique index uk_department_name (name);
```

查看表结构：

```sql
desc department;
```

如果 `name` 的 `Key` 显示：

```text
UNI
```

说明唯一约束已生效。

---

## 13. 一句话总结

```text
DTO 校验让错误更早发现，Service 校验表达业务规则，数据库约束保证最终数据底线。
```
