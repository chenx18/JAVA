# 03-分层架构

分层架构是后端项目最重要的结构基础。

它解决的问题是：不同代码各管各的，不要全堆在一个文件里。

---

## 最小分层

控制台练习阶段：

```text
Demo -> Service -> Repository -> List
```

Spring Boot 阶段：

```text
Controller -> Service -> Repository / Mapper -> List / Database
```

---

## Controller

Controller 是接口入口。

负责：

```text
接收请求
接收参数
调用 Service
返回结果
```

不建议负责：

```text
直接操作数据库
写复杂业务规则
写大量 if 校验
处理所有异常细节
```

---

## Service

Service 是业务层。

负责：

```text
业务规则
业务校验
调用 Repository / Mapper
决定抛什么业务异常
```

例如：

```java
Department department = mapper.findById(id);
if (department == null) {
  throw new DepartmentNotFoundException(id);
}
```

---

## Repository / Mapper

Repository / Mapper 是数据访问层。

内存版项目常见：

```text
Repository 操作 List
```

数据库版项目常见：

```text
Mapper 操作 MySQL
```

它们负责返回数据事实：

```text
查到了对象
没查到返回 null
修改/删除返回影响行数
```

---

## Model / Entity

Model / Entity 用来描述数据结构。

例如：

```java
public class Department {
  private int id;
  private String name;
}
```

它通常会作为：

```text
请求体对象
响应对象
数据库映射对象
```

---

## 为什么 Repository 通常不抛业务异常

Repository / Mapper 只知道数据访问结果。

它不应该决定业务含义。

```text
Mapper：id = 99 没查到
Service：没查到代表部门不存在，所以抛 DepartmentNotFoundException
```

一句话：

```text
数据层返回事实，业务层解释含义。
```

---

## 一句话总结

```text
Controller 管入口，Service 管业务，Repository / Mapper 管数据，Model 描述结构。
```
