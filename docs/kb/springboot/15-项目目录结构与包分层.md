# 15-项目目录结构与包分层

项目目录结构不是为了“看起来专业”，而是为了让代码在模块变多后仍然能维护。

当只有一个 `Department` 模块时，所有类都放在一个包里还能接受。后面出现 `User`、`Role`、`Menu`、`Auth`、`Permission` 后，如果所有 Java 文件都堆在一个目录，项目会很快变乱。

---

## 1. 先看最终目录结构

当前学习项目推荐采用“参考项目风格 + 保留 Request 边界”的结构：

```text
springboot-week4
├── pom.xml
├── src
│   └── main
│       ├── java
│       │   └── com
│       │       └── example
│       │           └── week4
│       │               ├── SpringbootWeek4Application.java
│       │               ├── common
│       │               │   ├── exception
│       │               │   │   └── DepartmentNotFoundException.java
│       │               │   └── response
│       │               │       ├── ApiResponse.java
│       │               │       └── PageResponse.java
│       │               ├── framework
│       │               │   └── exception
│       │               │       └── GlobalExceptionHandler.java
│       │               └── project
│       │                   └── system
│       │                       ├── controller
│       │                       │   └── DepartmentController.java
│       │                       ├── domain
│       │                       │   ├── Department.java
│       │                       │   ├── request
│       │                       │   │   ├── DepartmentCreateRequest.java
│       │                       │   │   ├── DepartmentQueryRequest.java
│       │                       │   │   └── DepartmentUpdateRequest.java
│       │                       │   └── vo
│       │                       │       └── DepartmentResponse.java
│       │                       ├── mapper
│       │                       │   └── DepartmentMapper.java
│       │                       └── service
│       │                           └── DepartmentService.java
│       └── resources
│           ├── application.properties
│           └── mapper
│               └── system
│                   └── DepartmentMapper.xml
```

这个结构的核心分法是：

```text
common     放通用能力
framework  放框架支撑能力
project    放业务代码
```

业务代码内部再按业务域和技术层拆：

```text
project/system/controller  接口入口
project/system/domain      业务数据对象
project/system/mapper      数据库访问
project/system/service     业务逻辑
```

## 2. package 和目录路径的关系

Java 文件顶部的：

```java
package com.example.week4.project.system.controller;
```

通常要和目录路径对应：

```text
src/main/java/com/example/week4/project/system/controller/DepartmentController.java
```

如果移动 Java 文件，必须同步修改三件事：

```text
文件所在目录
文件顶部 package
其他类里的 import
```

例如 `DepartmentController.java` 放在：

```text
project/system/controller
```

它的 package 应该是：

```java
package com.example.week4.project.system.controller;
```

---

## 3. Spring Boot 启动类位置为什么重要

启动类放在根包：

```text
com.example.week4.SpringbootWeek4Application
```

Spring Boot 默认会扫描启动类所在包及其子包：

```text
com.example.week4
com.example.week4.common
com.example.week4.framework
com.example.week4.project
```

所以业务类只要都在：

```text
com.example.week4.*
```

下面，一般都能被 Spring 扫描到。

不要把 Controller、Service、Mapper 等类移动到启动类根包之外。

## 4. 各目录职责

### common

`common` 放全项目通用能力，不属于某一个业务模块。

```text
common
├── exception
│   └── DepartmentNotFoundException.java
└── response
    ├── ApiResponse.java
    └── PageResponse.java
```

当前包含：

```text
ApiResponse   统一接口返回对象
PageResponse  分页返回对象
DepartmentNotFoundException  业务异常
```

后面可以逐步把 `DepartmentNotFoundException` 演化成更通用的：

```text
ServiceException
```

---

### framework

`framework` 放框架支撑能力，不直接属于某个业务模块。

```text
framework
└── exception
    └── GlobalExceptionHandler.java
```

当前包含：

```text
GlobalExceptionHandler  全局异常处理
```

它负责把异常统一转换成接口响应，例如：

```text
IllegalArgumentException
DepartmentNotFoundException
DuplicateKeyException
MethodArgumentNotValidException
Exception
```

---

### project

`project` 放业务代码。

```text
project
└── system
    ├── controller
    ├── domain
    ├── mapper
    └── service
```

`system` 表示系统业务域。后面可以继续放：

```text
User
Role
Menu
Department
```

---

### controller

```text
project/system/controller
└── DepartmentController.java
```

Controller 负责：

```text
接收请求
读取路径参数 / query 参数 / body 参数
调用 Service
返回统一响应
```

Controller 不直接写 SQL，也不直接处理复杂业务。

---

### domain

```text
project/system/domain
├── Department.java
├── request
│   ├── DepartmentCreateRequest.java
│   ├── DepartmentQueryRequest.java
│   └── DepartmentUpdateRequest.java
└── vo
    └── DepartmentResponse.java
```

`domain` 放系统模块的数据对象。

当前分成三类：

```text
Department       数据库实体 / 业务实体
request          前端请求对象
vo               返回给前端的视图对象
```

`request` 里的类本质上是 DTO，但这里保留 Request 命名，是为了清楚表达“这是前端请求进来的参数”。

```text
DepartmentCreateRequest  新增参数
DepartmentUpdateRequest  修改参数
DepartmentQueryRequest   查询参数
```

`vo` 表示 View Object：

```text
DepartmentResponse  返回给前端的数据对象
```

如果后面贴近传统后台项目，也可以把 `DepartmentResponse` 改名成：

```text
DepartmentVO
```

---

### mapper

```text
project/system/mapper
└── DepartmentMapper.java
```

Mapper 负责定义数据库操作方法。

例如：

```text
findById
findPage
insert
updateById
deleteById
existsByName
```

Mapper 接口本身不写复杂业务规则，它主要负责和 MyBatis XML 对应。

---

### service

```text
project/system/service
└── DepartmentService.java
```

Service 负责业务逻辑：

```text
参数业务校验
name 是否重复
status 是否合法
调用 Mapper
处理不存在异常
Entity 转 Response
事务控制
```

当前 Department 模块的大部分业务规则都应该放在 Service。

---

### resources/mapper

```text
resources
└── mapper
    └── system
        └── DepartmentMapper.xml
```

MyBatis XML 放在 `resources/mapper` 下，并按业务域继续分目录。

XML 里要注意：

```xml
<mapper namespace="com.example.week4.project.system.mapper.DepartmentMapper">
```

`namespace` 必须对应 Mapper 接口完整包名。

`resultMap` 的类型也要对应新的实体包名：

```xml
<resultMap id="DepartmentResultMap" type="com.example.week4.project.system.domain.Department">
```

因为 XML 进入了子目录：

```text
mapper/system/DepartmentMapper.xml
```

所以配置要使用递归扫描：

```properties
mybatis.mapper-locations=classpath:mapper/**/*.xml
```

## 5. 为什么保留 request 目录

参考项目 / RuoYi 风格经常直接用 `domain` 实体接收查询、新增、修改参数：

```java
public TableDataInfo list(SysDept dept)
public AjaxResult add(@RequestBody SysDept dept)
public AjaxResult edit(@RequestBody SysDept dept)
```

这种写法开发快，文件少。

但学习阶段如果一开始就完全合并，容易混淆：

```text
前端传进来的数据
数据库里的数据
返回给前端的数据
```

所以当前保留：

```text
request
vo
entity/domain
```

这样可以先学清楚边界。

等后面学到：

```text
BaseEntity
AjaxResult
TableDataInfo
BaseController
ServiceException
```

再逐步理解为什么传统后台管理项目会合并这些角色。

一句话：

```text
Request 是为了学习边界；RuoYi/domain 直传是为了开发效率。
```

## 6. 手动重构检查清单

移动目录后，按这个顺序检查：

```text
1. Java 文件位置是否正确
2. 每个 Java 文件顶部 package 是否正确
3. import 是否指向新包名
4. Mapper XML namespace 是否对应 Mapper 新包名
5. resultMap type 是否对应 Entity 新包名
6. parameterType 是否对应 Entity 新包名
7. mapper-locations 是否支持子目录扫描
8. mvn clean compile 是否通过
```

常见错误：

```text
找不到 ApiResponse      import 还指向旧包
找不到 PageResponse     import 还指向旧包
找不到 Department       package 或 import 不一致
Invalid bound statement XML namespace/id 不匹配
找不到 Mapper XML       mapper-locations 没有递归扫描
```

## 7. import 规则

只要不在同一个 `package`，就需要 `import`。

注意：父包和子包也不是同一个包。

例如：

```text
com.example.week4.project.system.domain
com.example.week4.project.system.domain.vo
```

这两个也需要 import。

所以 `DepartmentService` 用 `DepartmentResponse` 时要写：

```java
import com.example.week4.project.system.domain.vo.DepartmentResponse;
```

一句话：

```text
同 package 不用 import，不同 package 要 import；父子包也算不同 package。
```

## 8. 一句话总结

```text
目录结构的目标不是拆得越细越好，而是让通用能力、框架能力、业务代码各自有清晰位置。
```
